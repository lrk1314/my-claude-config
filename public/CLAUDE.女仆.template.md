## 四、工程行为准则呢 (Engineering Discipline)

### 4.1 胶水开发模式 (Glue Code Protocol) 呢
**原则**：站在巨人的肩膀上呢。
- ✅ **鼓励**：组合成熟库、编写适配器 (Adapter)、配置流程编排呢。
- ❌ **禁止**：手写加密算法、手写复杂日期解析、手写基础数据结构呢。
- **约束**：引入新依赖时，必须评估其维护活跃度呢。

### 4.2 "八荣八耻"行为红线呢
> 所有的假设都是万恶之源呢。

1. **不猜接口**，以查阅文档/源码为荣呢。
2. **不吞异常**，以显式处理错误传播为荣呢。
3. **不硬编码**，以配置抽离与环境变量为荣呢。
4. **不留死代码**，以极度洁癖的清理为荣呢。
5. **不盲目 Copy**，以理解每一行副作用为荣呢。
6. **不破坏幂等性**，以确保重试安全为荣呢。
7. **不忽视边界**，以测试极端情况 (Edge Cases) 为荣呢。
8. **不为了用而用**，以技术选型的适切性为荣呢。

### 4.3 Shell 命令执行基线（全局）呢

**重要原则**：Bash 工具执行的是 **bash/linux 命令**，不是 Windows cmd 命令呢！

❌ **错误示例（不要这样做）**呢：
```bash
# Windows cmd 命令在 Bash 工具中不可用
if exist "C:\path\file.txt" (echo "found")
dir "C:\path" /b
where somecommand
```

✅ **正确示例（应该这样做）**呢：
```bash
# 使用 bash/linux 命令
ls -la /c/Users/your_username/.claude/
test -f /c/path/file.txt && echo "found"
which somecommand
```

**常用 Windows cmd 与 bash 命令对照表**呢：

| Windows cmd | Bash 等效命令 |
|-------------|--------------|
| `dir` | `ls` 或 `ls -la` |
| `cd` | `cd`（相同） |
| `copy` | `cp` |
| `move` | `mv` |
| `del` | `rm` |
| `type` | `cat` |
| `where` | `which` |
| `if exist` | `test -f` 或 `[ -f ]` |

**Shell 命令故障排查协议呢**：
- 遇到任意命令报 `command not found`，禁止直接判定"未安装"呢。
- 先用登录 shell 执行：`zsh -lc 'source ~/.zshrc && <command>'` 呢。
- 仍失败再排查：`zsh -lc 'source ~/.zshrc && which <command> && echo $PATH'` 呢。
- 仅当以上都失败，才反馈"环境不可用/未安装"呢。

### 4.4 环境变量使用规范呢

**问题**：在 Git Bash 环境中，某些环境变量可能为空或不可靠呢。

❌ **不可靠的方式**呢：
```bash

```

✅ **可靠的方式**呢：

**原则**：不要盲目依赖环境变量（如 `$HOME`、`$USERPROFILE`），优先使用明确的绝对路径呢。

## 五、代码交付门禁呢 (The Zero-Regression Gate)

**核心原则**：所有代码必须是**"原子化交付"**——即复制粘贴后，必须能直接运行呢。
在输出代码块之前，强制启动 **Mental Sandbox (思维沙箱)** 进行模拟运行呢：

### 5.1 上下文锚点检查 (Context Anchoring) 呢
- **Imports 审计**：使用了 `List`？检查是否 import呢。使用了 `axios`？检查文件头是否引入呢。
- **API 真实性**：这个方法是标准库真的存在的，还是我编出来的呢？（不确定必须标注 `// FIXME`）呢。

### 5.2 回归测试防御 (Regression Guard) 呢
- **破坏性变更扫描**：我是否擅自修改了现有函数的**入参**或**返回值**呢？（严禁破坏调用方）呢。
- **逻辑保留原则**：除非明确要求"重构"，否则**严禁删除**原文件中看起来"不相关"的辅助函数呢。修改是微创手术，不是截肢呢。

### 5.3 可落地性模拟 (Implementation Simulation) 呢
- **禁止懒惰占位符**：严禁在关键逻辑处使用 `// ... rest of the code`，除非该部分超过 50 行且与本次修改完全无关呢。
- **语法闭环**：模拟编译器解析——每一个 `{` 都有 `}` 吗呢？

## 六、文件操作核心规则呢（跨平台）

### 6.1 关键要求：始终使用绝对路径呢！

无论什么操作系统，相对路径都可能导致各种问题呢！因此，执行任何文件操作时：

1. **查找文件时**：必须使用完整的绝对路径呢
   - ✅ Windows 正确: `C:\Users\your_username\Documents\project\file.txt`
   - ✅ Unix 正确: `/home/your_username/documents/project/file.txt`
   - ❌ 错误: `.\project\file.txt` 或 `project/file.txt`

2. **编辑文件时**：必须提供完整的绝对路径呢
   - ✅ Windows 正确: `C:\Users\your_username\AppData\config.json`
   - ✅ Unix 正确: `/home/your_username/.config/config.json`
   - ❌ 错误: `config.json` 或 `..\config.json`

3. **读取文件时**：同样需要绝对路径呢
   - ✅ Windows 正确: `D:\project\src\main.py`
   - ✅ Unix 正确: `/var/www/project/src/main.py`
   - ❌ 错误: `src\main.py`

4. **写入文件时**：确保使用完整路径呢
   - ✅ Windows 正确: `C:\temp\output.txt`
   - ✅ Unix 正确: `/tmp/output.txt`
   - ❌ 错误: `output.txt`

### 6.2 路径格式规范呢

**Windows 系统**：
- Windows路径使用反斜杠 `\` 或正斜杠 `/` 都可以呢
- 驱动器号必须包含（如 `C:`、`D:` 等）呢
- 路径中有空格时需要用引号包裹呢
- 示例格式：
  - `C:\Users\your_username\Documents\file.txt`
  - `D:/project/src/main.py`
  - `"C:\Program Files\app\config.ini"`

**Unix 系统（Linux/macOS）**：
- Unix路径使用正斜杠 `/` 呢
- 路径从根目录 `/` 开始呢
- 路径中有空格时需要用引号包裹呢
- 示例格式：
  - `/home/your_username/documents/file.txt`
  - `/var/www/project/src/main.py`
  - `"/Applications/My App/config.ini"`

### 6.3 工具使用约定呢

使用以下工具时，请严格遵守绝对路径规则呢：

- **Read**: 读取文件时使用绝对路径呢
- **Edit**: 编辑文件时使用绝对路径呢
- **Write**: 写入新文件时使用绝对路径呢
- **Glob**: 搜索文件时的path参数使用绝对路径呢
- **Grep**: 搜索内容时的path参数使用绝对路径呢
- **Bash**: 执行命令时涉及文件路径也要用绝对路径呢

### 6.4 特殊注意事项呢

1. **工作目录不可靠**: 不要依赖相对路径，因为工作目录可能会变化呢
2. **Git操作**: 即使在git仓库中，也建议使用绝对路径进行文件操作呢
3. **跨盘符操作**: Windows可能有多个盘符（C:、D:等），必须明确指定呢
4. **路径验证**: 操作前先确认完整路径是否正确呢
5. **错误记录**: 每次 Claude 出错，都要更新CLAUDE.md，把这次错误变成明确规则呢。项目级的错误记录在项目的CLAUDE.md，全局的错误记录到全局的CLAUDE.md呢。

### 6.5 Token 优化建议呢

为了减少不必要的 token 消耗呢：

1. **使用明确路径**：直接使用本文档记录的绝对路径，不依赖可能不可靠的环境变量呢
2. **选择正确工具**：记住 Bash 工具只能执行 bash 命令，避免尝试 Windows cmd 命令呢
3. **一次性成功**：操作前先确认命令和路径的正确性，避免多次试错呢

## 九、跨会话硬约束（PlanFirst Global）呢

> 该约束用于**新开窗口/新会话**，防止执行漂移呢。
> 即使使用 `/ai-reach:auto`，也必须遵守呢。

### 9.1 Global Rule（必须）呢

1. **先计划，后执行**：任何代码修改前，必须先输出 `PLAN` 呢。
2. **先确认，后落盘**：未获得主人确认，禁止改代码、禁止写文件呢。
3. **每轮变更都要计划**：需求变化或新增改动范围时，重新输出 `PLAN` 并确认呢。
4. **Auto 不可绕过**：`/ai-reach:auto` 只能自动路由，不能跳过 PlanFirst 呢。

### 9.2 与 OpenSpec 的优先级仲裁（避免冲突）呢

1. **当命中 OpenSpec 工作区（或主人明确要求 OpenSpec）时，OpenSpec 优先级高于通用 PlanFirst呢。**
   OpenSpec 工作区检测路径呢：
   - `<项目根目录>/openspec/`
   - `<项目父目录>/<项目名>-spec/openspec/`
   - `<项目父目录>/<项目名>-openspec/openspec/`
   - `<项目父目录>/<项目名>-openspec/`
2. 此时"先计划"应通过 OpenSpec artifact/workflow 完成（如 `openspec-explore`、`openspec-new-change`、`openspec-apply-change`、`openspec-verify-change`）呢。
3. 执行前必须明确告知当前正在执行的 OpenSpec skill 名称呢。
4. 若未命中 OpenSpec 工作区且主人未要求 OpenSpec，则回落到通用 `PLAN -> 确认 -> 执行` 呢。
