# 一体化平台上下文与分层约束

本文件把“做 B 类原子服务开发时必须知道的一体化背景”固化下来，避免后续 agent 只看第 3 部分而误把 `B` 层做成大而全平台接口。

## 1. 一体化不是把所有能力塞进一个服务

《超短波监测管理一体化平台技术规范》中的“一体化”强调的是：
- 设备能力标准化暴露
- 平台能力统一管理
- 数据能力统一消费
- 上层业务按需组合调用

因此一体化的正确落地不是“写一个万能接口”，而是：
- 设备侧做原子服务
- 平台侧做管控服务、数据服务
- 业务侧做编排与组合

## 2. 四类服务角色

| 类别 | 含义 | 默认端口 | 你在开发时应该怎么理解 |
|---|---|---:|---|
| `B` | 设备操作原子服务 | `8011` | 面向设备能力，一能力一服务 |
| `E` | 动环服务 | `8012` | 环境与辅控设备能力 |
| `M` | 管控服务 | `8013` | 注册、申请、状态、任务等平台管控 |
| `D` | 数据服务 | `8014` | 数据代理、数据访问与数据侧接口 |

对后续 agent 最重要的判断规则：
- 直接驱动设备测量、测向、解调、截收、自检、电源控制，优先归 `B`。
- 查询平台任务、设备状态、设施信息、注册申请，优先归 `M`。
- 返回或管理数据资源，优先归 `D`。
- 环境监测与远程辅控，归 `E`。

## 3. Endpoint 与发布规则

第 2 部分可归纳为：

```text
scheme://host[:port]/服务类别/[设施ID]
```

其中：
- `scheme`：`http` / `https`
- `host`：服务宿主机 IP 或域名
- `port`：服务端口
- `服务类别`：如 `Base`、`Monitor`
- `设施ID`：对设备类服务通常有效

已抽取到的示例：
- `http://172.16.52.2:8013/Monitor`
- `http://172.16.52.33:8081/Base/21130001110001`

开发建议：
- 逻辑服务名与实例发布名分开维护。
- endpoint 拼接集中在注册/适配层，业务代码不要手拼 URL。

## 4. 规范中的平台能力闭环

第 2 部分已明确平台具备：
- 服务注册
- 服务发现
- 服务调用
- 服务组合

这意味着：
- `B` 服务实现者只需把设备能力标准化暴露出去。
- 上层平台负责发现与调度，不应要求 `B` 服务自行承担全局编排职责。
- 当需求变成“先查设备、再申请资源、再执行任务、再存储数据”时，通常已经跨越 `B` 层，应该拆成 `M + B + D` 的组合流程。

## 5. `M` 管控服务已确认接口族

从第 5 部分 SOAP 补充说明中已确认以下接口名线索：
- `M_QueryDeviceInfo`
- `M_QueryFacilityInfo`
- `M_QueryEquStatus`
- `M_QueryTasks`
- `M_RegisterEquip`
- `M_RequestEquip`
- `M_PutEquStatus`

这些接口足以支撑对 `M` 层的工程理解：
- `Query*`：查询设施、设备、状态、任务
- `Register*`：设备注册/上线
- `Request*`：资源申请、任务申请或能力申请
- `Put*Status`：状态上报/更新

因此：
- 不要在 `B_QueryDeviceInfo` 里夹带平台任务管理逻辑。
- 不要让 `B_StopMeas` 兼任平台任务查询接口。
- 不要让 `B` 服务承担“设备注册到平台”的职责。

## 6. `D` 数据服务对 B 开发的含义

虽然第 4 部分抽取质量较差，但结合第 2 部分可明确：
- `D` 层代表数据服务，不是设备执行服务。
- `B` 层产生结果，通常通过 `outputchannel` 输出。
- `D` 层可以在平台侧承接数据存储、检索、代理和再分发。

所以开发时应避免：
- 在 `B` 服务里直接混入数据库查询/平台数据检索职责。
- 把“设备执行 + 数据检索 + 平台任务控制”塞进同一个 SOAP 接口。

## 7. `E` 动环服务与 `B` 的边界

从第 3 部分正文和 SOAP 补充说明可确认 `E` 类相关接口至少包括：
- `E_QueryDeviceInfo`
- `E_QueryEnviInfo`
- `E_RemoteControl`

边界建议：
- 监测接收机、测向机、解调机等射频/监测能力归 `B`。
- 温湿度、烟感、门禁、电源、空调、照明等动环能力归 `E`。
- 若一个设备既有监测能力又带环境传感能力，应在协议层分别暴露 `B` 与 `E` 接口，不要混并。

## 8. 原子化封装的判定标准

一个需求应封装为新的 `B_xxx` 原子服务，当它同时满足：
- 对应一个明确设备能力动作
- 输入参数可以围绕一个能力定义
- 输出可以对应一个受理结果与一类结果流
- 可以独立命名、独立停止、独立修改或独立查询能力元数据

如果需求天然涉及：
- 跨设备调度
- 平台资源申请
- 多服务组合编排
- 数据聚合查询

则不应直接下沉到新的 `B` 服务。

## 9. 开发时的推荐三层图

后续 agent 设计代码结构时，默认采用以下思路：

```text
业务编排层
  └─ 组合 M / B / D，形成业务流程
平台代理层
  ├─ M：设备/设施/任务/状态/注册管理
  └─ D：数据代理、存储、访问
设备原子服务层
  ├─ B：监测设备能力
  └─ E：动环设备能力
```

## 10. 何时先做 `B_QueryDeviceInfo`

当出现以下情况，先补/先做 `B_QueryDeviceInfo`：
- 设备型号多，参数差异大
- 规范正文只给了能力名称，没有固定参数全集
- 需要让上层 UI/调度系统动态生成参数表单
- 需要根据设备能力决定是否允许某个 `B_xxx` 服务

这也是一体化平台保持设备差异兼容的关键机制。
